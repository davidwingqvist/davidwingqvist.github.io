Multithreading

Homehearth uses a technique called Double buffering to ensure smooth data transition between the rendering thread and the update thread. When the render thread is ready for a new batch of objects to render, it tells the update thread that a new batch is requested, waits for update thread to provide with the new data, and then renders the data.

Thread pooling

I've also implemented a technique called 'thread pooling' which differs from simple thread creation by creating threads and putting them in a state of waiting for job requests. When a thread discovers a job requests, it grabs it from the queue and commits to the job task. Afterwards it releases the results and returns to its passive wait state. This technique would potentially save performance from the heavy task of creating and removing threads each frame.

Thread Code

As shown in the picture to the right, when doing a thread pool, we need to create a "main"-function for these threads. In this main function, we setup the state of several needed components and then put the threads in a 'while'-loop where they will wait for work.

To avoid any issues with racing and deadlocking, we utilize mutexes and locks to control who gets to grab next job from the queue.

When work is done, the compiled draw commands are sent to the main thread to be later sent to the GPU.
